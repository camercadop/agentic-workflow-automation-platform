# ADR 008 – Agent-Assisted Development Model

**Status:** Accepted
**Date:** 2026-06-19
**Authors:** Carlos Mercado <carlos.mercado@example.com>

## Context
The platform's evolution requires consistent, high-quality artifact generation (plugins, ADRs, tests, documentation) while maintaining architectural integrity. Manual processes lead to inconsistencies, omissions, and architectural drift. A structured agentic approach can systematize development activities without introducing runtime complexity.

## Terminology
| Term | Definition |
|------|------------|
| **Development Agent** | An autonomous software entity that assists engineering tasks. Executes in development environments, not the runtime platform. |
| **Architect Agent** | Specialized agent responsible for architectural decisions, ADR generation, and design validation. |
| **Plugin Generator Agent** | Agent that produces compliant plugins following contract specifications. |
| **Test Generator Agent** | Agent that creates unit, integration, and contract tests for generated or existing code. |
| **Documentation Agent** | Agent that produces and maintains platform documentation. |
| **Subagent** | Temporary specialized development agent delegated by another development agent to complete a bounded engineering task. Subagents do not exist outside the development process. |
| **Skill** | Reusable capability or tool that an agent can invoke during development tasks. |

## Decision
Adopt an Agent-Assisted Development Model where **development-time agents** collaborate to produce platform artifacts. Agents operate in the software development lifecycle, not the runtime architecture.

```
Developer
    ├─ Architect Agent
    │    ├─ Plugin Generator Agent
    │    │    └─ Test Generator Agent
    │    └─ Documentation Agent
    └─ Review Agent (optional)
```

Generated artifacts become part of the platform:
- Plugins (ADR-005 compliant)
- ADRs (ADR-001 through ADR-007 compliant)
- Tests
- Documentation

## Relationship with Runtime Architecture
Agents are **explicitly separate** from the runtime architecture defined in ADRs 001–007. In particular, components such as the **Validation Engine** described in ADR‑009 are **build‑time** services that run during CI/CD and are **not** part of the runtime platform. This reinforces the principle that development agents operate only in development environments and never become runtime components.

| Runtime Component | Development Agent |
|-------------------|-------------------|
| Plugins | Generated by Plugin Generator Agent |
| Workflows | Designed with Architect Agent input |
| Execution Contexts | Not created or managed by agents |
| Plugin Contracts | Enforced by Architect Agent verification |
| Manifest Handling (ADR-002) | Agents generate compliant manifests |
| Lifecycle (ADR-003) | Agents produce lifecycle-compliant code |
| Isolation (ADR-004) | Agents ensure generated code respects boundaries |

Agents never become runtime components.

## Artifact Generation Flow
1. **Architect Agent** analyzes requirements and generates ADRs
2. **Plugin Generator Agent** produces plugins adhering to contracts
3. **Test Generator Agent** creates validation tests
4. **Documentation Agent** produces user guides and API references
5. **Review Agent** validates artifacts against platform standards

## Agent Collaboration Model
- Agents communicate via shared artifact repositories and design documents
- Subagents are spawned for specialized tasks (e.g., code generation vs. documentation)
- Skills are reusable tools (linters, formatters, validators) invoked by agents
- All agents maintain traceability to source decisions and requirements

## Governance and Quality Controls
- **Contract Compliance**: All generated plugins must satisfy Plugin Contract Model (ADR-005)
- **Manifest Generation**: Plugins include ADR-002 compliant manifests
- **Test Coverage**: Minimum coverage enforced by Test Generator Agent
- **Documentation**: All public APIs documented via Documentation Agent
- **Architectural Review**: Changes validated against existing ADRs

## Validation Criteria
- Generated plugins compile and pass all tests
- New ADRs follow established format and reference relevant existing ADRs
- Test suites achieve mandatory coverage thresholds
- Documentation is complete and accurate

## Alternatives Considered
- **Manual Development Process**: Rejected—too slow and error-prone for rapid iteration.
- **Separate AI Tooling Outside Architecture**: Rejected—loses integration with architectural governance.
- **Runtime Agent Integration**: Rejected—introduces runtime complexity and security concerns.

## Mandatory Rules
- Agents must generate artifacts compliant with existing ADRs
- Generated code must pass all validation checks
- Agents cannot modify runtime architecture definitions
- All agent-generated artifacts must be human-reviewable

## Allowed Changes
- Extend agent skill sets with new development tools
- Add new agent types for emerging development needs
- Modify artifact generation templates

## Forbidden Changes
- Deploying agents as runtime components
- Bypassing architectural governance for generated artifacts
- Creating agent-managed execution contexts or workflows
